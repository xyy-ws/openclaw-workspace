## 2) 遇到的问题（按类型）

### A. 权限与凭据问题
- 现象：GitHub API 可用但 push/PR 出现 403 或认证失败。
- 根因：Token 作用域与仓库访问范围不完整；子进程环境无凭据。
- 解决：补齐权限（Contents + PR 等），主流程使用一次性认证 push 补推。

### B. 预览稳定性问题
- 现象：`502 Bad Gateway`、`404`、Vite host 拦截。
- 根因：前端 dev 进程被杀；子路径部署未统一 base/basename；allowedHosts 未配置。
- 解决：
  - 重启 dev 服务；
  - Vite `base=/habit/` + Router `basename=/habit`；
  - `server.allowedHosts` 加域名；
  - Nginx `/habit/` 反代到 5173。

### C. 速度体感问题
- 现象：用户感知“进展慢”。
- 根因：先做完整链路与测试，缺少“短周期可见成果”。
- 解决：切为“20~45 分钟可见交付”节奏（先视觉样片再全量重构）。

### D. 质量问题
- 现象：E2E 超时、Vitest 异步清理警告。
- 根因：Vite React 插件/渲染链路问题；组件异步状态更新时机。
- 解决：系统化调试（先复现再定位根因）+ 最小修复 + 回归验证。

## 3) 推荐方法（下次照着走）

采用 **KPT + RCA + SOP** 三段式：

### KPT（Keep / Problem / Try）
- Keep：需求澄清、任务拆解、分支化推进、实时 progress。
- Problem：权限/部署反复、反馈节奏不稳定。
- Try：先出视觉样片，再并入主项目；认证前置检查脚本。

### RCA（Root Cause Analysis）
对每个问题固定记录 4 行：
- 现象
- 根因
- 修复
- 防复发措施

### SOP（标准流程）
把下方“开工清单”固定为模板，开新项目直接复用。

## 4) 下次最佳开工清单（强烈建议）

### 4.1 启动前（10 分钟）
- [ ] 确认目标：MVP 的唯一北极星指标
- [ ] 确认交付节奏：每 30~60 分钟至少一个可见里程碑
- [ ] 确认 GitHub：仓库、分支、token 权限、PR 权限
- [ ] 确认预览：域名路径、Nginx、端口、allowedHosts

### 4.2 开发中
- [ ] 每个任务：先失败测试 → 实现 → 回归
- [ ] 每个任务：push + 更新 `docs/progress.md`
- [ ] 每 2~3 个任务：产出可视演示（截图/可访问 URL）

### 4.3 UI 项目专用
- [ ] 默认提供素材层（图标/插画/空状态），避免纯文字页面
- [ ] 先做“单页样片”确认风格，再全量迁移
- [ ] 动效只用 transform/opacity，兼容 `prefers-reduced-motion`

### 4.4 收尾
- [ ] 单测/构建/E2E 全绿
- [ ] 发布说明（改了什么、怎么试、已知问题）
- [ ] 复盘入库（本文件）

## 5) 本次经验结论（TL;DR）
- **先给可见效果，再做完整性**，用户体感会好很多。
- **部署与认证前置检查**能省掉大量返工。
- **测试驱动 + 分步提交**虽然前期慢一点，但后期稳定且可追溯。
